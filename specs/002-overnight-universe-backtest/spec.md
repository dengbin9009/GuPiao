# 功能规格说明：一夜持股法股票池选股与真实分钟线回测增强

**功能分支**：`002-overnight-universe-backtest`  
**创建日期**：2026-06-25  
**状态**：Draft  
**输入**：增强现有内置策略「一夜持股法」，让候选股选择从“自选股第一只”调整为“扫描配置股票池后筛选”，并补充一个可复现的隔夜分钟线回测脚本。该脚本必须优先使用本地缓存，只有在缓存覆盖不足时才允许在线补拉数据。

## 用户场景与测试

### 核心用户故事

作为管理员，我希望内置的一夜持股法能够从配置好的 A 股股票池中自动挑选候选股，而不是受自选股顺序影响；同时我还希望有一个基于真实分钟线数据的最近两日隔夜回测脚本，这样我才能判断策略真实会选哪只股票，以及一笔隔夜交易在扣除费用和滑点之后到底能赚多少钱。

### 验收场景

1. **当** 一夜持股法在模拟盘模式下运行，且行情数据与事件数据都处于新鲜状态时，**如果** 开始执行候选股评估，**那么** 系统必须扫描配置股票池中的合格 A 股，而不能依赖自选股顺序。
2. **当** 有多只股票同时通过一夜持股法筛选条件时，**如果** 系统完成候选股排序，**那么** 必须选择排名第一的股票进入下单流程，其余通过筛选但未被选中的股票要记录为候选溢出或未选中候选。
3. **当** 自选股中某只股票的排名低于另一只符合条件的股票时，**如果** 策略运行，**那么** 系统必须优先选择排名更高的股票，即使它不是自选股中的第一只。
4. **当** 没有任何股票通过股票池筛选条件时，**如果** 策略运行结束，**那么** 此次运行必须以“无下单”完成，并明确记录接受/拒绝摘要。
5. **当** 管理员运行最近两日隔夜回测脚本并指定股票与日期范围时，**如果** 本地缓存的分钟线已经完整覆盖所需的入场和出场窗口，**那么** 脚本必须只使用本地缓存并输出净收益结果。
6. **当** 本地分钟线缓存缺失或覆盖不完整时，**如果** 回测脚本运行，**那么** 脚本必须尝试在线拉取数据、更新缓存，并在重新校验覆盖范围后再继续计算。
7. **当** 在线拉取后仍然无法获得完整的分钟线覆盖时，**如果** 回测脚本运行，**那么** 脚本必须以可操作的缺失数据错误失败，且绝不能回退到演示用的生成 K 线。
8. **当** 回测脚本成功计算出一笔隔夜交易时，**如果** 输出结果，**那么** 结果中必须包含入场/出场时间、入场/出场价格、成交数量、毛收益、手续费、印花税、滑点影响、净收益和收益率。

### 边界情况

- 股票池中存在停牌、退市整理、ST、涨跌停不可交易或行情已过期的股票。
- 多只股票的排序关键字段完全相同。
- 分钟线数据覆盖了入场日，但没有覆盖次日出场时间。
- 指定的入场日和出场日之间存在节假日或非交易日。
- 行情数据是新鲜的，但公司事件数据已经过期。
- 在线数据提供方返回了不完整的分钟线或时间戳异常的数据。
- 本地缓存文件存在，但不包含入场窗口或出场窗口内的关键分钟线。

## 需求

### 功能需求

- **FR-001**：一夜持股法的候选股选择必须从持久化的股票主数据中扫描配置好的 A 股股票池，不能再默认使用自选股顺序。
- **FR-002**：自选股必须继续保留为观察、手动刷新和管理功能，但不能再作为一夜持股法默认的候选股来源。
- **FR-003**：股票池扫描必须遵守现有一夜持股法的过滤条件，包括交易所范围、ST 排除、可交易性、上市时长、成交额、换手率、日内涨幅、日内 VWAP、均线过滤、基准过滤和事件风险过滤。
- **FR-004**：候选股的通过/拒绝判断必须复用现有一夜持股法的过滤和排序逻辑，确保手动运行、定时运行和测试辅助流程使用同一套规则。
- **FR-005**：最终入场股票必须是排序后第一个通过筛选的候选股；其余通过筛选但未被选中的股票必须出现在策略日志或运行摘要中。
- **FR-006**：策略运行日志或摘要必须包含扫描候选总数、通过数量、拒绝数量、最终选中股票代码，以及在未下单时的主要拒绝原因。
- **FR-007**：如果没有任何候选股通过筛选，此次策略运行必须以“完成但不下单”结束，并明确记录该结果，而不是回退到任何硬编码股票。
- **FR-008**：系统必须提供一个独立的后端脚本，用于执行单只股票的最近两日隔夜分钟线回测。
- **FR-009**：该回测脚本必须支持配置股票代码、入场日期、出场日期、初始资金、佣金费率、最低佣金、印花税率、过户费率、滑点基点、入场时间和出场时间。
- **FR-010**：回测脚本在请求分钟线数据时，必须优先读取项目中的市场数据缓存文件，再决定是否触发在线拉取。
- **FR-011**：如果本地分钟线缓存对指定交易窗口的覆盖不完整，脚本必须尝试在线拉取数据、合并到缓存，并在重新校验覆盖后再继续执行。
- **FR-012**：如果在线拉取后仍然无法覆盖所需分钟线，脚本必须以清晰的缺失数据错误终止，且不能使用演示生成 K 线替代。
- **FR-013**：回测脚本在计算净收益时，必须应用 A 股一手 100 股、佣金、最低佣金、卖出印花税、过户费和配置滑点规则。
- **FR-014**：回测脚本必须按照可复现的规则，从配置的尾盘入场窗口中选择入场分钟线，并从次日出场窗口中选择出场分钟线；该规则必须在代码和输出中可说明。
- **FR-015**：回测脚本必须同时输出机器可读的结果摘要，以及便于人工快速查看的控制台结果摘要。
- **FR-016**：本次增量范围不能要求重做现有完整回测 UI，也不能要求整体替换当前历史回测框架。

### 非功能需求

- **NFR-001**：股票池选股增强不能破坏现有模拟盘和真实盘的安全前置检查，包括 stale quote 和 stale event 的 fail-closed 行为。
- **NFR-002**：最近两日隔夜回测脚本在相同分钟线缓存和相同参数条件下，必须产生可重复的结果。
- **NFR-003**：回测脚本在缺失数据时必须显式失败，不能用静默降级方式掩盖数据覆盖不足的问题。

## 关键实体

- **UniverseCandidate（股票池候选项）**：在一夜持股法过滤前，由股票主数据、实时行情和可交易性字段组合而成的内存态候选股对象。
- **OvernightSelectionResult（隔夜选股结果）**：记录某次一夜持股法评估中最终选中股票、通过候选和拒绝候选详情的结果对象。
- **MinuteBarCoverageReport（分钟线覆盖报告）**：描述本地缓存与在线补拉后，是否完整覆盖指定入场和出场窗口的数据校验结果。
- **RecentOvernightBacktestResult（最近隔夜回测结果）**：记录单笔隔夜回测的入场/出场时间、价格、数量、费用、毛收益、净收益和收益率的结果对象。

## 成功标准

- **SC-001**：当股票池中存在两只及以上符合条件的股票时，一夜持股法会选择排名最高的候选股，而不是自选股中的第一只。
- **SC-002**：仅改变自选股顺序，不改变市场数据和股票池数据时，一夜持股法的选股结果不应发生变化。
- **SC-003**：最近隔夜回测脚本能够仅基于缓存分钟线数据计算净收益，并且不使用任何演示生成 K 线。
- **SC-004**：当本地缓存覆盖不足时，回测脚本会尝试通过已配置数据源补拉一次，并在失败时报告具体缺失时间窗口。
- **SC-005**：完成本次选股逻辑增强后，原有 stale data 风控仍然对一夜持股法自动运行持续生效。

## 澄清说明

- **2026-06-25**：本次增量将一夜持股法从“受自选股顺序影响的选股方式”调整为“基于配置股票池扫描后的选股方式”。
- **2026-06-25**：自选股仍然保留，用于观察、刷新和人工管理，但不再默认决定一夜持股法的选股目标。
- **2026-06-25**：最近两日隔夜回测能力以脚本形式提供，本次不扩展为完整的回测 UI 改造。
- **2026-06-25**：最近两日隔夜回测必须优先读取本地缓存，只有在缓存覆盖不足时才允许在线补拉。
- **2026-06-25**：在线补拉后仍缺失分钟线数据时，必须硬失败，明确禁止回退到演示生成 K 线。

## 不在本次范围内

- 重做现有完整历史回测引擎或回测页面。
- 一次回测多只股票的组合隔夜收益。
- 新增独立的股票池范围管理页面。
- 放宽 stale quote 或 stale event 的 fail-closed 风控要求。
