# GuPiao Loop 约束

> 每轮开始必须读取本文件。以下规则均为强制约束，不能被优先级、预算或修复目标覆盖。

## 第一周模式

- 第一周固定为 L1 `report-only`：只检查、报告并更新 Loop 文档，不自动修改业务代码。
- 不创建 Codex Automation；所有循环由人工显式启动。
- 进入代码修复前必须获得人工批准。

## 分支与工作区

- 代码修复只能在专用 git worktree 中进行，禁止在 `main` 分支修改。
- 一次只处理一个最小问题，不覆盖、撤销或整理其他代理/用户的改动。
- 未经人工明确批准，不得 push、merge、deploy、发布、关闭问题或标记 PR ready。

## 密钥与敏感文件

- 禁止读取、编辑、提交、复制或输出 `.env`、`.env.*` 及其中任何内容。
- 禁止输出密钥、令牌、密码、账户标识、券商凭据或真实订单信息。
- 只能通过规格、示例文件、代码默认值或无敏感信息的进程参数验证配置；不能用读取 `.env` 的方式验证。

## 交易安全

- `LIVE_TRADING_ENABLED` 必须保持 `false`，不得启用或建议循环自动启用。
- `BROKER_ADAPTER` 必须保持 `simulation`，不得改为其他适配器。
- 禁止连接或调用真实券商、解锁真实账户、发送真实订单或模拟真实订单探测。
- 任何涉及实盘配置、账户、适配器或下单的事项必须立即停止并请求人工确认。
- 即使获得人工确认，循环也不得自动执行最终真实交易动作。

## 代码与验证

- 采用 TDD：先复现失败，再做最小修复，最后运行相关测试和全套验收。
- 禁止删除、跳过、弱化测试或隐藏异常来制造通过结果。
- 修复前后都要检查作用域，只改当前问题需要的文件。
- 同一错误最多尝试 3 次，第三次失败后熔断并升级人工。
- 单任务最多 6 次迭代；不得重命名错误或拆分循环规避限制。
- 完成标准以 `LOOP.md` 的全部二进制退出条件为准。

## 预算与停止

- 第一周每轮最多 0 个子代理，每日最多运行 1 次，最多 50k token。
- 达到预算 80% 后只报告；达到 100% 或检测到 `loop-pause-all` 后立即退出。
- 发现实盘风险、密钥暴露风险、非 worktree 修改或 `main` 分支修复时立即停止并升级人工。

## 沟通

- 执行动作前说明将做什么；结束时报告证据、不确定项和人工门。
- 不得宣称未验证的条件已经通过。
